## Agent Z strategic lessons

### 1. Ask-size discipline

The median non-cancelled ZCG project size is about $45k. Category medians are more useful than overall averages:

- Community: ~$13.4k median;
- Event sponsorship: ~$15.9k median;
- Education: ~$34.4k median;
- Research & Development: ~$37k median;
- Infrastructure: ~$51k median;
- Integration: ~$69.6k median;
- Wallets: ~$128.5k median.

Agent Z should use this to challenge over-large requests. A first-time regional, community, or education proposal asking for six figures needs unusually strong justification.

### 2. Large grants require credibility

The largest grants are not random. They cluster around protocol extensions, wallets, integration, infrastructure, and known ecosystem teams.

Agent Z should not encourage applicants to copy big QEDIT/Tor/Zingo-style budgets unless they have comparable technical depth, public-good importance, and delivery credibility.

### 3. Proposal status history shows high rejection/filtering risk

With 295 rejected proposals vs 153 approved proposals in the all-grants tracker, Agent Z should behave like a pre-submission reviewer, not a cheerleader.

Default behaviour should be:

- identify likely objections;
- reduce scope;
- convert vague impact into milestones;
- force Zcash-specific alignment;
- warn when a proposal is not ready.

### 4. Forum procedure is a real gate

Coinholder tracking includes proposals not considered because the forum post was missing or the submission was late. Recent ZCG GitHub issue comments also repeatedly require forum links.

Agent Z should treat forum readiness as a binary gate, not a soft recommendation.

### 5. Milestone liabilities matter

ZCG currently has about $1.65m in future grant liabilities and $1.0m in future security bounty liabilities in this workbook. Agent Z should understand that ZCG can fund new work, but proposals compete against already-approved commitments and should justify why now.

### 6. Community/education grants are common but smaller

Community is the largest project-count category, but its median grant size is low compared with wallets/infrastructure/protocol categories. Agent Z should guide non-technical applicants toward smaller, measurable pilots rather than broad long-term asks.

### 7. Technical categories can support larger asks

Wallets, infrastructure, integration, and protocol-extension grants have higher ceilings. But these proposals need:

- architecture;
- dependencies;
- test plan;
- security model;
- maintenance plan;
- user validation;
- upstream merge or integration plan.

## Agent Z rules to encode

1. If requested amount is above the category median by more than 2x, require stronger justification.
2. If request is above $50k, flag KYC and stronger due diligence.
3. If community/education/event proposal is above $50k, recommend a smaller pilot unless proof of reach is strong.
4. If forum link is missing, mark proposal as not review-ready.
5. If milestones do not equal total request, mark proposal as not review-ready.
6. If proposal does not map to a ZCG category, mark fit as weak.
7. If applicant has no delivery evidence, reduce recommended budget and add proof requirements.
8. Use the dashboard as financial benchmark evidence, not as the only source of truth.
9. Treat dashboard as stale when status says updates required or block/time is older than current date.
10. Separate ZCG grant evidence from Coinholder grant evidence because their processes and status mechanics differ.
